# 个人知识库

一个用于保存、整理和复用资料的个人知识库模板。

使用时可以把资料直接发给 Codex，然后告诉它“加入到知识库”。Codex 会负责把原始资料保存到 `raw/`，再整理成可检索、可追溯、可问答的 `wiki/` 知识页。

## 这个目录是做什么的

这个目录是一套轻量知识库框架，用来把零散资料沉淀成长期可用的知识系统。

它适合保存和整理：

- 文章、网页、PDF、文档
- 会议纪要、聊天记录、访谈记录
- 临时想法、项目笔记、复盘内容
- 工具使用说明、操作流程、经验总结
- 方案、案例、研究资料、参考资料

它的核心思路很简单：

- `raw/` 保存原始资料，尽量不改事实。
- `wiki/` 保存整理后的知识页，方便阅读、问答和复用。
- `_system/` 保存给 Codex 使用的模板、提示词和校验脚本。
- `context/` 保存项目背景、长期说明、补充上下文。

## 它解决了什么问题

- 资料太散：资料不用分散在聊天、网盘、浏览器收藏夹和本地文件里。
- 找不到来源：每条重要结论都可以回到原始资料。
- 看过就忘：资料会被整理成主题、概念、流程和对比页。
- 难以复用：一次经验可以变成以后能继续问、继续改、继续用的知识。
- 整理成本高：把资料给 Codex，Codex 会帮你放进知识库。

## 目录结构

```text
个人知识库/
├─ context/                  # 背景说明、长期上下文、项目约束
├─ raw/                      # 原始资料层，保存证据和原文
├─ wiki/                     # 知识页层，保存整理后的内容
│  ├─ 00-index/              # 知识库导航和总览
│  ├─ 10-source/             # 每份资料对应的来源页
│  ├─ 20-topic/              # 某个主题的综合整理
│  ├─ 30-concept/            # 重要概念、术语、定义
│  ├─ 40-process/            # 操作步骤、工作流、方法
│  └─ 50-comparison/         # 多个方案、观点、工具之间的对比
└─ _system/
   ├─ prompts/               # Codex 整理知识库时使用的提示词
   ├─ scripts/               # 知识库校验脚本
   └─ templates/             # raw/wiki 页面模板
```

## 使用方式

打开 Codex，在这个目录里直接发消息即可。

### 加入一份资料

你可以这样说：

```text
把这份资料加入到知识库。
```

然后直接粘贴文章、会议纪要、聊天记录、想法、网页内容，或者提供本地文件路径。

例如：

```text
把 D:\tmp\某个文档.pdf 加入到知识库，并整理成可以复用的 wiki 页面。
```

```text
下面是一次会议纪要，帮我加入到知识库：

（粘贴会议内容）
```

```text
这是我刚想到的一个项目思路，加入知识库，并帮我整理成主题页和后续问题：

（粘贴想法）
```

### 更新已有知识

如果你已经有相关内容，可以这样说：

```text
把下面的新资料加入知识库，如果已有相关主题，就合并更新，不要重复建页。
```

Codex 会优先检查 `wiki/` 里有没有相近主题，再决定是新建页面还是更新已有页面。

### 基于知识库提问

你可以直接问：

```text
基于这个知识库，帮我总结一下某个主题的核心结论。
```

```text
基于 raw 和 wiki，告诉我这个方案有哪些风险和下一步行动。
```

```text
帮我找出知识库里和“用户留存”相关的内容，并整理成行动建议。
```

### 让 Codex 做维护

你也可以让 Codex 检查知识库：

```text
检查这个知识库有没有断链、缺字段或结构问题。
```

```text
帮我整理最近新增的 raw，把能合并的内容合并到已有 wiki 页面。
```

```text
帮我给这个知识库生成一个索引页。
```

## Codex 会做什么

当你说“加入到知识库”时，Codex 通常会做这些事：

1. 阅读你给的资料。
2. 判断资料类型和主题。
3. 在 `raw/` 中保存原始资料或原始资料摘要。
4. 在 `wiki/10-source/` 建立来源页。
5. 在 `wiki/20-topic/`、`wiki/30-concept/`、`wiki/40-process/` 或 `wiki/50-comparison/` 中建立或更新知识页。
6. 给重要结论加来源链接。
7. 标记不确定内容，不编造。
8. 运行校验脚本，检查知识库结构是否正常。
9. 告诉你新增或更新了哪些文件。

## 常用指令

### 入库资料

```text
把这份资料加入到知识库，保留原始信息，并整理成 wiki 页面。
```

### 整理主题

```text
基于知识库，整理一个“主题名”的主题页，包含定义、核心结论、关键概念、使用场景和风险。
```

### 生成流程

```text
把这些零散记录整理成一个可执行流程页，写清楚步骤、输入、输出和注意事项。
```

### 合并重复内容

```text
检查知识库里是否有重复主题，把重复内容合并，并保留来源。
```

### 做问答

```text
只基于这个知识库回答我的问题，不确定的地方请明确说不确定。
```

### 做复盘

```text
基于这些 raw 和 wiki，帮我输出一份复盘，包含做了什么、有效经验、问题、下一步行动。
```

## 页面类型说明

页面类型由 Codex 根据资料内容判断。下面是各目录的用途：

| 目录 | 用途 |
| --- | --- |
| `wiki/00-index/` | 知识库入口、导航、总览 |
| `wiki/10-source/` | 某份资料、文件、文章、记录的来源页 |
| `wiki/20-topic/` | 围绕一个主题的综合整理 |
| `wiki/30-concept/` | 某个概念、术语、模型、定义 |
| `wiki/40-process/` | 可执行步骤、操作手册、工作流 |
| `wiki/50-comparison/` | 多个工具、方案、观点、版本之间的比较 |

## 知识库原则

- 原始资料先保留，再整理。
- 整理后的结论要尽量能追溯到来源。
- 不确定的内容要标记，不要猜。
- 重复资料尽量合并，不要到处建相同页面。
- 低价值资料不用删除，可以标记为低价值。
- `wiki/` 可以持续更新，知识库不是一次性文档。

## 校验知识库

如果你会使用命令行，可以运行：

```powershell
powershell -ExecutionPolicy Bypass -File _system/scripts/validate-kb.ps1
```

严格校验：

```powershell
powershell -ExecutionPolicy Bypass -File _system/scripts/validate-kb.ps1 -StrictSchema
```

也可以直接对 Codex 说：

```text
帮我运行知识库校验，并修复能自动修复的问题。
```

## 推荐工作流

基础工作流：

```text
把资料发给 Codex
  -> 说“加入到知识库”
  -> Codex 保存 raw 并整理 wiki
  -> 你继续基于知识库提问或更新
```

更完整的工作流：

```text
收集资料
  -> 发给 Codex
  -> Codex 入库 raw
  -> Codex 生成或更新 wiki
  -> Codex 检查断链和缺字段
  -> 你基于知识库做问答、复盘、写方案或生成手册
```

## 示例：从一段聊天记录开始

你可以直接这样发给 Codex：

```text
把下面聊天记录加入到知识库，提炼成主题页。如果里面有可执行步骤，也帮我建一个流程页。

（粘贴聊天记录）
```

Codex 应该输出：

- 新增了哪份 raw
- 新增或更新了哪些 wiki 页面
- 提炼出的核心结论
- 还需要你确认的问题

## 示例：从一个 PDF 开始

你可以直接这样发：

```text
把 D:\tmp\资料.pdf 加入到知识库。请先提取内容，再生成来源页、主题页和必要的概念页。
```

如果 PDF 内容很长，可以继续说：

```text
继续处理剩余页面，并把所有结论合并到同一个主题下。
```

## 示例：让 Codex 帮你找资料

```text
基于这个知识库，找出所有和“项目复盘”相关的内容，并按主题整理。
```

```text
基于这个知识库，帮我生成一份“我目前有哪些可复用方法”的清单。
```

## 维护建议

- 有资料就直接发给 Codex，不要先纠结分类。
- 每次入库后，让 Codex 告诉你新增和更新了哪些页面。
- 一段时间后，让 Codex 帮你合并重复主题。
- 重要主题可以让 Codex 生成索引页。
- 如果知识库变大，定期让 Codex 运行校验脚本。
